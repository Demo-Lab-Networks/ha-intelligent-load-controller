# Traceability Matrix

**Status:** Phase 0 canonical traceability index
**Source notation:** `S§x` is the numbered section of the supplied product brief.
**State:** `Planned` means implementation evidence is intentionally not yet produced; it is not a pass.

## Usage

Every source requirement has a stable ID. Test IDs are defined in `TEST_PLAN.md`; implementation artifacts are added when the relevant phase lands. Each row is one independently verifiable obligation. New behaviour must add/update a row before merge. A requirement is only `Verified` when its linked automated evidence and required manual/release artifact exist.

The **Implementation area / accountable owner** is an accountable subsystem or workstream, not a named individual or a claim that the work is complete. A **Planned acceptance artifact** is the future evidence expected before the row can be verified; it is deliberately not proof that the artifact exists today.

| Owner label | Accountable implementation area |
| --- | --- |
| Foundation | Packaging, manifests, local distribution, tooling, CI, and repository baseline. |
| Policy | Pure controller models, scheduling, tariffs, planning, safety rules, and accounting primitives. |
| HA backend | Config flows, schema, lifecycle, persistence, entities, actions, WebSocket API, diagnostics, and repairs. |
| Adapters | Home Assistant/device boundary, actuator semantics, and feedback handling. |
| Orchestration | Site arbitrator, capacity allocation, fairness, staggering, and multi-load coordination. |

| Safety | Cross-cutting no-blind-action, constraint, fault, and recovery guarantees. |
| Architecture | ADRs, compatibility decisions, and future-interface boundaries. |
| Documentation | User/developer/governance documentation and published evidence. |
| Release | Final packaging, external publication prerequisites, and release evidence assembly. |
